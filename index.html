The W3C oversee the development of Web technology standards

The W3C have defined a universe of rules that govern the relationship between all the CSS properties and how they are visually formatted. The fundamental rules of the Visual Formatting Model act as a point of departure for this discussion.

The Visual Formatting Model

The visual formatting model is an algorithm that processes a document and displays it on visual media. During this process, each element within the document tree generates zero or more boxes, according to the box model.

A box is rendered by the model in relation to the edge of its containing block. Once rendered, a box establishes itself as a containing block for its descendants.

Visual Formatting Model:
Box Generation

Different types of boxes can be generated depending on the value of the display CSS property. The box type affects how the visual formatting is performed.

When the value of a box's CSS display property is declared, the element is said to be at block-level.

.flex-wrapper { display: flex; }

Layout Mode

A CSS layout mode, sometimes abbreviated as layout, is an algorithm determining the position and the size of boxes based on the way they interact with their sibling and ancestor boxes. There are several of them:

The table layout The block layout The inline layout The flexible box layout The grid layout

Positioning schemes

After the boxes are generated, they are positioned.

  • the static / relative positioning scheme.
  • the absolute / fixed positioning scheme removes elements entirely from the document flow.

The Flow

The normal flow positions each box one after the other. The floats algorithm extracts the box from the normal flow and put it to the side of the containing box.

Elements belonging to the flow content category typically contain text or embedded content.

The normal flow is triggered:

.normal-flow { position: relative / static; float: none; }

Combining these fundamental principles into a class

The properties of this containing block create a solid foundation from which to control the position, style and structure of the internal child elements.

.containing-block { box-model: border-box; position: relative; display: flex; }
Visual Format Containing block Box Model

A containing block that uses a positioning scheme, layout mode and the border-box model, create rules which contain and allow more control over child elements.

B_E__M - is this an efficient methodology ?

"Let's say we need to modify the appearance of one of the library blocks. That doesn't require changing the CSS rules of the block in the library source code or copying the code at the project level. We only need to create additional CSS rules for that block at the project level."

"Blocks can be moved around on a page, moved between pages or projects. Functional Naming Methodology

The other extreme of the naming continuum, is defined by utility based naming conventions such as Attribute Modules for CSS (AMCSS) and approaches that are function based. Functional approaches allow components to be built using more classes that apply one or two properties at one time to form a multi-class syntax. This approach does not create a dependency between parent and child elements. Child elements can exist in isolation.

How does one implement this approach?

There are two answers to this question.

Firstly, one could copy the functional classes used within this presentation or duplicate the classes from this website by Jon Gold .

Secondly, one could author their own function based stylesheet.

Authoring a function based stylesheet

For every property there is a class.

For every property value, there is a class.

A function based stylesheet could contain thousands of classes, with each class being a slight variation of another. The type and quantity of function based classes within a stylesheet is determined by the front-end architectural context it is used in.

Examples of function based classes
